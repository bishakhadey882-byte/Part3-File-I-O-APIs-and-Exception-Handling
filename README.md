**Theme: Product Explorer & Error-Resilient Logger**

## 📌 Project Overview

This project is a Python application that:
- Reads and writes student notes to files using proper file handling modes
- Fetches real product data from the **DummyJSON public API** (no auth required)
- Filters, sorts, and searches products programmatically
- Handles all failure scenarios gracefully using structured exception handling
- Logs errors with timestamps to a persistent `error_log.txt` file

All four tasks are implemented in a single well-commented Python file: `part3_api_files.py`

## ✅ Task 1 — File Read & Write Basics 

**Location:** Top of `part3_api_files.py`

### Part A — Write:

Opens `python_notes.txt` in **write mode (`'w'`)** with `encoding="utf-8"` and writes 5 lines:

```
Topic 1: Variables store data. Python is dynamically typed.
Topic 2: Lists are ordered and mutable.
Topic 3: Dictionaries store key-value pairs.
Topic 4: Loops automate repetitive tasks.
Topic 5: Exception handling prevents crashes.
```

Prints: `File written successfully.`

Then opens the same file in **append mode (`'a'`)** and adds 2 custom lines:

```
Topic 6: Functions help reuse code and improve readability.
Topic 7: Modules allow us to organize and import code easily.
```

Prints: `Lines appended.`

> The actual `python_notes.txt` in this repo confirms all 7 lines are present.

### Part B — Read:

1. Reads all lines and prints each one **numbered** (strips trailing `\n` using `.strip()`):
   ```
   1. Topic 1: Variables store data. Python is dynamically typed.
   2. Topic 2: Lists are ordered and mutable.
   ...
   7. Topic 7: Modules allow us to organize and import code easily.
   ```
2. Prints total line count: `Total number of lines in the file: 7`
3. Asks the user for a keyword and prints all **matching lines (case-insensitive)**. If no match is found, prints:
   `No lines found containing the keyword '<keyword>'.`

## ✅ Task 3 — Exception Handling 

**Location:** Lower section of `part3_api_files.py`

### Part A — Guarded Calculator (`safe_divide`):

```python
def safe_divide(a, b):
```

- Returns `a / b` on success
- Catches `ZeroDivisionError` → returns `"Error: Cannot divide by zero"`
- Catches `TypeError` → returns `"Error: Invalid input types"`

Test results:
```
safe_divide(10, 2)     → 5.0
safe_divide(10, 0)     → Error: Cannot divide by zero
safe_divide("ten", 2)  → Error: Invalid input types
```

### Part B — Guarded File Reader (`read_file_safe`):

```python
def read_file_safe(filename):
```

- Tries to open and read the file, returns full content as a string
- Catches `FileNotFoundError` → prints `"Error: File '{filename}' not found."`
- Uses a `finally` block to **always** print `"File operation attempt complete."` — whether file was found or not

Test results:
```
read_file_safe("python_notes.txt")  → succeeds, prints content
read_file_safe("error_log.txt")    → prints error + finally message
```

### Part C — Robust API Calls:

All Task 2 API calls are refactored into functions (`fetch_products`, `fetch_laptops`, `post_product`) with every `requests.get()` and `requests.post()` wrapped in `try-except` that handles:

| Exception | Message Printed |
|---|---|
| `requests.exceptions.ConnectionError` | `Connection failed. Please check your internet.` |
| `requests.exceptions.Timeout` | `Request timed out. Try again later.` |
| `Exception as e` | Prints the error message |

All requests use `timeout=5`.

### Part D — Input Validation Loop:

A `while True` loop that:
1. Prompts: `"Enter a product ID to look up (1-100), or 'quit' to exit: "`
2. Validates input — must be an integer in range 1–100; prints a warning and re-asks if not (no API call is made with invalid input — uses `continue`)
3. Makes GET request to `https://dummyjson.com/products/{id}`
4. `status_code == 404` → prints `"Product not found."`
5. `status_code == 200` → prints product `title` and `price`
6. Continues until user types `quit`

## ✅ Task 4 — Logging to File 

**Location:** Bottom of `part3_api_files.py`

### How the Logger Works:

A reusable `log_error(function_name, error_type, message)` function:
- Opens `error_log.txt` in **append mode (`'a'`)** so logs accumulate across runs
- Uses `datetime.now().strftime("%Y-%m-%d %H:%M:%S")` to timestamp each entry
- Writes entries in the required format:
  ```
  [YYYY-MM-DD HH:MM:SS] ERROR in function_name: ErrorType — message
  ```

### Two Logged Entries Demonstrated:

**Entry 1 — ConnectionError:**
Intentionally requests a genuinely unreachable URL inside a `try-except`:
```python
requests.get("https://this-host-does-not-exist-xyz.com/api", timeout=5)
```
Caught as `ConnectionError` and logged under function name `fetch_products`.

**Entry 2 — HTTP 404 Error:**
Requests product ID `999` (does not exist on DummyJSON). Since a 404 is **not a Python exception**, it is detected by checking `response.status_code != 200` inside an `if` block and logged manually under `lookup_product`.

> **Key concept demonstrated:** HTTP errors (like 404) are server responses — they do NOT raise Python exceptions. They must be handled by checking `response.status_code`, not with `try-except`.

### Actual `error_log.txt` Contents (included in repo):

```
[2025-01-15 14:32:01] ERROR in fetch_products: ConnectionError — No connection could be made
[2025-01-15 14:33:10] ERROR in lookup_product: HTTPError — 404 Not Found for product ID 999
```

After logging both entries, the script reads and prints the full contents of `error_log.txt` to confirm the logger is working correctly.
